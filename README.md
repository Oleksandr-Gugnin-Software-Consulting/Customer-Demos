# Customer-Demos — презентация CI / GitHub Actions

Этот репозиторий — демонстрация CI-конфигурации, подготовленная для просмотра потенциальными клиентами и работодателями. Это не подробное учебное пособие: цель — показать архитектуру, практические приёмы и возможности, которые команда может воспроизвести в продакшн-проектах.

Коротко о том, что именно демонстрируется и почему это может быть интересно заказчику:

- **Архитектура CI на уровне команды** — основной workflow, который оркеструет переиспользуемые блоки (reusable workflows) для тестов, линта и сканирования безопасности.
- **Переиспользуемые workflow-файлы** — показывает, как разделить CI на независимые, повторно используемые компоненты (unit, integration, lint, typecheck, security, performance).
- **Парадигма матриц** — запуск тестов на нескольких версиях Python и сценариях окружения, чтобы гарантировать совместимость.
- **Интеграционные прогонки с Docker Compose** — демонстрация запуска вспомогательных сервисов (Postgres/Redis и т.д.) для realistic integration tests.
- **DevOps-механики** — пример сборки Docker-образа, проверки артефактов и контроль размера образа.
- **Self-hosted runner deployment** — скрипт и compose-файлы для развертывания собственных раннеров (демонстрация, как можно управлять инфраструктурой CI вне GitHub-hosted раннеров).

Ключевые файлы и папки (быстрая навигация):

- `/.github/workflows/ci.yml` — основной pipeline, который вызывает переиспользуемые блоки и управляет условиями запуска.
- `/.github/workflows/reusable-*.yml` — переиспользуемые workflow-файлы для отдельных задач.
- `/.github/ci/runners/` — docker-compose файлы и вспомогательные скрипты для self-hosted раннеров.
- `/.github/ci/runners/deploy_runner.sh` — утилита для автоматического развёртывания раннера и генерации регистрационных токенов через GitHub CLI.
- `pytest.ini`, `pyproject.toml`, `setup.py` — файлы, облегчивающие установку и прогон тестов в CI.

Что важно понимать при просмотре демо:

- Этот репозиторий подготовлен для демонстрации практических решений, акцент на архитектуре CI и рабочих паттернах, а не на детальном обучении каждого инструмента.
- В ряде мест использованы pragmatic-правки (например, временные заглушки), чтобы прогон CI в демо был надёжным. В реальном проекте эти элементы стоит заменить на более строгие проверки готовности сервисов и на полноценную конфигурацию зависимостей.

Как оценить демо (рекомендации для просмотрщика):

- Посмотрите `/.github/workflows/ci.yml` — это «карта» процесса CI.
- Откройте один из `reusable-*.yml` — чтобы увидеть, как вынести логику джоба в переиспользуемый блок.
- Обратите внимание на `/.github/ci/runners/` — демонстрация готовности развернуть и поддерживать self-hosted раннеры.
